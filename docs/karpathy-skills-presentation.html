The original project

What the original skill did

The original repository packaged a compact set of behavioral rules for coding agents, mainly around Claude Code and Cursor.

Think Before Coding

Do not silently assume. Surface ambiguity, tradeoffs, and confusion before implementation.

Simplicity First

Prefer the smallest correct solution. Avoid speculative abstractions and unrequested flexibility.

Surgical Changes

Touch only what the task requires. Avoid drive-by refactors and style drift.

Goal-Driven Execution

Turn vague requests into success criteria and loop until the result is verified.

Karpathy's methodology

The failure modes it targets

The methodology comes from a practical observation: LLM coding agents are powerful, but they fail in predictable ways unless the working loop is constrained.

Wrong assumptions

Agents often pick an interpretation silently and run with it.

Overengineering

They add layers, options, wrappers, and abstractions before they are needed.

Drive-by edits

They change nearby code, comments, or formatting that was not part of the task.

The countermeasure is not a bigger prompt. It is a disciplined loop: clarify, minimize, change surgically, verify.

The operating model

Principles become a workflow

1

Clarify

State assumptions. Ask only when guessing would materially change the result.

→
2

Minimize

Choose the smallest implementation that solves the requested problem.

→
3

Touch only the code needed for the task. Match local style.

→
4

Verify

Run the narrowest meaningful test, build, typecheck, lint, or formatter.

What I changed

I added the missing feedback loop

The original principles said “keep it simple.” The improvement turns that into an explicit post-code habit.

Before

Simplicity was a principle the agent should remember while coding.

After

After generating code, the agent must do a brief simplification pass before finalizing.

That means removing unnecessary wrappers, dead code, speculative flexibility, and extra concepts introduced by the change — while preserving behavior.

Code example 1

Skill and Claude guidance now include simplification

# CLAUDE.md and skills/karpathy-guidelines/SKILL.md

Ask yourself: "Would a senior engineer say this is overcomplicated?"
If yes, simplify.

+ After generating or modifying code, run a brief simplification pass before finalizing:
+ - Remove unnecessary abstractions, wrappers, dead code, or speculative flexibility introduced by the change.
+ - Check whether the same behavior can be expressed with fewer concepts or less code.
+ - Keep only simplifications that preserve the requested behavior and make the diff easier to review.

This is the main non-Amp improvement: “simplicity first” becomes a concrete self-review checkpoint.

Code example 2

The project now includes an Amp-native AGENTS.md

+ ## 5. Post-Code Simplification Pass
+
+ After generating or modifying code, always perform a brief simplification pass before finalizing:
+
+ 1. Remove unnecessary abstraction, wrappers, dead code, or speculative flexibility introduced by the change.
+ 2. Prefer the smallest correct implementation.
+ 3. Check for duplication introduced by the change.
+ 4. Ensure the code follows this repository's existing patterns and AGENTS.md instructions.
+ 5. Run the narrowest relevant verification: focused test, typecheck, lint, formatter, or build.
+ 6. Mention what was simplified, or state that no meaningful simplification was needed.

This lets Amp apply the same discipline automatically at the project-instruction level.

Code example 3

The README is no longer Claude-only

- # Karpathy-Inspired Claude Code Guidelines
+ # Karpathy-Inspired Coding Agent Guidelines

- A single CLAUDE.md file to improve Claude Code behavior...
+ A small set of reusable guidance files to improve coding-agent behavior in Claude Code, Amp, and Cursor...

The positioning moves from one tool to a portable coding-agent quality package.

The Ampcode angle

Why Amp benefits from this

AGENTS.md as always-on behavior

Amp reads project guidance automatically, so the principles become default behavior instead of a manual reminder.

Skills for on-demand focus

The same guidelines can live as an Amp skill for explicit review, simplification, or refactoring tasks.

Verification fits Amp's loop

The guidance pairs naturally with Amp's autonomous implementation + targeted verification workflow.

Smaller, safer diffs

It reinforces Amp's best behavior: understand enough, change only what matters, verify, then stop.
